# 代码审查规则（Code Review Rules）

> 项目：**adenexus.com**  
> 技术栈：Next.js 14 · React 18 · Tailwind CSS 3.4 · TypeScript  
> 适用范围：所有前端 PR（页面 / 组件 / 样式 / API Route）

***

## 一、审查基本原则（必须认同）

1.  **品牌一致性 > 页面完成速度**
2.  **可维护性 > 一次性实现**
3.  **是否方便下一个人接手，是首要标准**
4.  Code Review 的目标是 **收敛，而不是展示技术能力**

***

## 二、PR 基本要求（不满足直接打回）

### ✅ PR 必须包含

*   清晰的 PR 描述（改了什么 / 为什么改）
*   对应页面或组件的截图（PC + Mobile）
*   如涉及样式：说明是否使用了既有组件

### ❌ PR 不可包含

*   未使用的组件 / 文件
*   被注释掉的大段代码
*   与本 PR 无关的重构

***

## 三、目录与职责审查清单（必查）

### ✅ 页面（`app/`）

*   是否只负责**组合 Section**
*   是否未引入 UI 细节逻辑
*   是否遵循 Server / Client 边界

❌ 拒绝：

*   页面中直接写复杂样式逻辑
*   页面中定义可复用组件

***

### ✅ Section（`components/sections/`）

*   是否由 `Section + Container` 组成
*   是否只关注布局与结构
*   是否未直接访问 API / 表单逻辑

***

### ✅ UI 组件（`components/ui/`）

*   是否是**可复用、无业务语义**
*   是否未使用 Framer Motion
*   是否样式完全来自 Tailwind tokens

***

## 四、样式审查规则（最重要）

### 4.1 颜色使用（强制）

✅ 允许：

```tsx
text-aden-orange
bg-aden-dark-blue
border-aden-mid-grey
```

❌ 直接打回：

```tsx
text-[#ED4F18]
bg-[#283948]
style={{ color: "#ED4F18" }}
```

> **任何 HEX / RGB / inline style 一律拒绝**

***

### 4.2 字体使用（必查）

✅ 合法：

```tsx
font-title
font-body
font-cn
```

❌ 不允许：

```tsx
font-sans
font-serif
```

***

### 4.3 布局宽度（必查）

✅ 合法：

```tsx
<Container>...</Container>
```

❌ 拒绝：

```tsx
<div className="w-[1180px] mx-auto px-[32px]">
```

***

## 五、组件使用审查（强约束）

### 5.1 Button

✅ 必须使用：

```tsx
<Button variant="primary" />
<Button variant="dark" />
<Button variant="outline" />
```

❌ 不允许页面自造按钮样式：

```tsx
<button className="bg-aden-orange px-6 py-3 text-white">
```

***

### 5.2 Card / Section

*   Service / Feature / Block **必须使用 Card**
*   Section 背景色 **必须通过 `tone` 控制**

❌ 页面中直接造“临时盒子”一律拒绝

***

## 六、Framer Motion 审查规则（严格）

✅ 允许：

*   Section 级动画
*   `FadeIn` 统一封装

❌ 直接拒绝：

*   Button / Input 动画
*   UI 组件中使用 motion
*   Server Component 中使用 motion

> 原则：**动画是装饰，不是结构**

***

## 七、Next.js 14 / React 审查点

### ✅ Server / Client 边界

*   页面与 Section **默认 Server**
*   `use client` 仅用于：
    *   表单
    *   动画
    *   交互组件

❌ 滥用 `use client` 直接拒绝

***

### ✅ Metadata / SEO

*   新页面是否包含 `metadata`
*   是否破坏 i18n 路由结构

***

## 八、表单 & API 审查规则

### ✅ 表单

*   是否使用 React Hook Form
*   是否使用 Zod schema
*   是否无手写校验逻辑

### ✅ API Route

*   是否只负责请求处理
*   是否未包含业务 UI 逻辑
*   是否与前端共享 schema

***

## 九、TypeScript 与代码质量

✅ 必须：

*   无 `any`
*   Props 明确定义
*   不忽略 ESLint 错误

❌ 拒绝：

*   `// @ts-ignore`
*   为“省事”关闭规则

***

## 十、Code Review 快速打分法（实操）

**评审人只需回答 5 个问题：**

1.  是否使用了品牌 tokens？
2.  是否复用了已有组件？
3.  是否让代码更容易维护？
4.  是否破坏既有结构？
5.  是否方便下一个人继续开发？

> **只要有 1 个是 “否” → 不合并**

***
