# 技术栈选型说明

**Technology Stack Decision Record**

> 项目：**adenexus.com 官方网站**  
> 文档目的：解释并固化本项目的前端技术栈选型依据  
> 适用对象：管理层 / 前端团队 / 外包 / 未来维护人员

***

## 1. 文档背景

本项目为 **企业级官方网站**，而非一次性营销页或后台系统。  
其核心特征包括：

*   多语言（英语 / 法语 / 德语 / 阿拉伯语）
*   明确的品牌规范与视觉一致性要求
*   表单交互（联系表单）
*   SEO 与合规页面
*   预计 **长期维护与持续演进**

因此，本项目在技术选型时，**优先考虑“稳定性、可治理性与长期成本”**，而非最轻量或最流行方案。

***

## 3. 核心选型原则（先说“为什么”）

本项目技术栈遵循以下 **5 条选型原则**：

1.  **官网优先，而非产品型应用**
2.  **品牌一致性 > 开发灵活性**
3.  **治理能力 > 单点性能最优**
4.  **长期维护成本优先**
5.  **为未来扩展预留空间，但不提前过度复杂化**

***

## 4. 核心技术选型说明

### 4.1 为什么选择 Next.js 14（而不是 Astro / 纯 SSG）

**选择原因：**

*   支持 **Server Components**，利于 SEO 与性能
*   内建 API Routes，避免额外后端服务
*   对 i18n、表单、合规页面支持成熟
*   适合“内容 + 交互 + 少量后端”的官网形态

**未选方案的原因：**

*   Astro / 纯 SSG 更适合“静态内容站”
*   在多语言、表单、API 方面会引入额外复杂度

✅ **结论：Next.js 是“企业官网的稳妥选择”，而非过度设计。**

***

### 4.2 为什么使用 Tailwind CSS（而不是 SCSS / CSS Modules）

**选择原因：**

*   设计 Token 可工程化（颜色 / 字体 / spacing）
*   与组件基线（Button / Card / Section）天然契合
*   在 Code Review 阶段可强约束样式滥用
*   避免样式文件分散与长期失控

**前提条件（已满足）：**

*   明确的品牌规范
*   严格的开发约定
*   禁止 inline style 与 HEX 颜色

✅ **结论：在强治理前提下，Tailwind 是“最可控”的方案。**

***

### 4.3 TypeScript 的使用边界说明

本项目使用 TypeScript 的目标是：

*   提供**清晰的接口边界**
*   避免运行期错误
*   提升长期维护安全性

但 **不追求复杂类型体操**：

*   不滥用泛型
*   不引入高复杂度类型工具链
*   使用 Zod 作为运行期与编译期的桥梁

✅ **这是官网项目中“性价比最高”的 TS 使用方式。**

***

### 4.4 Framer Motion 的选型与约束

**为什么选：**

*   API 成熟
*   与 React 深度集成
*   适合做“区块级动效”

**为什么要强约束：**

*   官网动画的目标是 **节奏引导**
*   而不是视觉炫技

因此明确：

*   ✅ 只用于 Section 级
*   ❌ 不用于 Button / Input
*   ❌ 不用于 Server Components

✅ **选型正确，治理更关键。**

***

### 4.5 表单方案（RHF + Zod）

**选择原因：**

*   性能好、社区成熟
*   表单状态与校验解耦
*   可实现前后端 schema 统一

即使当前只有 Contact Form，  
该选型也为未来复杂表单预留空间，而不增加当前复杂度。

✅ **这是“不会后悔”的表单选型。**

***

## 5. 关于“是否过度设计”的说明

从短期视角看：

*   治理文档偏多
*   组件约束较严格

但从长期视角看：

*   官网是品牌资产
*   一致性比速度更重要
*   技术债务一旦产生，极难回收

✅ **这是“健康的前置设计”，而非浪费。**

***

## 6. 技术栈的适用边界（非常重要）

本技术栈 **适用于**：

*   企业官网
*   品牌站
*   多语言内容站
*   带轻交互与表单的站点

本技术栈 **不适用于**（需另行选型）：

*   内部后台管理系统
*   高度交互的 SaaS 应用
*   数据密集型 Dashboard

***

## 7. 最终结论（可对外引用）

> **本项目技术栈并非追求最轻或最新，  
> 而是在当前官网定位下，  
> 选择了“最稳健、最可治理、最不容易后悔”的方案。**
